# 專案執行規範

## 長任務規範

當使用者明確說出「使用長任務規範」、「啟動長任務」、「長任務模式」或 `$long-task` 時，必須載入並遵守 [`docs/long-task-protocol.md`](docs/long-task-protocol.md)。

此規範讓代理人自行完成「檢查 → 實作 → 驗證 → 修正 → 再驗證」循環；不得為一般測試、診斷、格式化、型別檢查、建置或安全的本機修正要求使用者確認。

一般 `git commit` 與 `git push` 也屬自主流程；只有網路、認證、遠端權限或其他規範列出的受保護動作，才可以停下來等待使用者明確確認。

## 分支規範

- 每次開發新功能或修正 BUG 前，都必須從目前基準分支另開專用分支；不得直接在 `main` 或其他共用分支上修改。
- 新功能分支使用 `feature/<簡短描述>` 命名，BUG 修正分支使用 `bugfix/<簡短描述>` 命名。
- 開始修改前先確認目前分支；若尚未位於對應的專用分支，代理人必須先建立並切換到該分支，再進行實作。
