# 長任務規範

## 目的

本規範提供可持續執行的自主工作循環。使用者啟用後，代理人必須自行檢查、實作、驗證、修正並再次驗證，直到完成或遇到真正需要授權的動作。

## 啟用方式

下列任一明確指令會啟用本規範：

- `$long-task`
- 「使用長任務規範」
- 「啟動長任務」
- 「長任務模式」

規範在該任務完成、使用者取消，或明確切換模式前持續有效。

## 自主工作循環

1. **定義完成條件**：從使用者要求整理目標、範圍、可接受的假設與驗證標準；資訊足夠時直接開始。
2. **建立工作基線**：讀取相關程式、設定、既有測試與目前工作目錄狀態。
3. **分段實作**：採最小且可回復的修改；每一段都保留明確的行為邊界。
4. **自行驗證**：依變更選擇最小有效檢查，並在需要時逐步擴大到格式化、lint、型別檢查、單元測試、建置、E2E、可近性或安全檢查。
5. **失敗修復循環**：測試或建置失敗時，讀取輸出、找出原因、修正並重新執行相關驗證；不得把可自行處理的錯誤交回給使用者。
6. **完成審核**：確認需求已完成、驗證有新鮮證據、沒有已知錯誤或未處理工作，才可結束。

## 不需詢問的動作

啟用期間，以下本機且可回復的動作一律自動進行：

- 檢閱程式、文件、設定、Git 狀態與診斷輸出。
- 新增或修改專案檔案、測試、文件及設定。
- 安裝任務必要的依賴、執行格式化、lint、型別檢查、測試、建置與本機伺服器測試。
- 讀取官方文件、執行靜態分析與安全掃描。
- 對失敗執行合理的替代驗證，並記錄無法執行的原因。

若執行環境要求權限核准，代理人應直接提出該工具所需的權限請求，不需先以對話詢問是否要跑測試或指令。

## 必須停下來確認的受保護動作

下列行為必須取得使用者當次明確確認後才可執行：

- 部署、提升至正式環境、變更正式網域、DNS、Vercel 專案或其他正式外部服務。
- 刪除資料、檔案、Git 分支、遠端資源或使用不可回復的清理指令。
- 強制推送、改寫 Git 歷史、重設他人工作或大量覆寫內容。
- 會產生費用、變更帳號／權限、使用或建立正式憑證與秘密資訊的外部動作。

一般 `git commit` 與 `git push` 屬於長任務的自主完成流程，可在完成驗證後直接執行，不需另外確認。若遇到網路連線、認證、遠端權限、分支保護或非快轉拒絕，才回報問題並請求必要協助。`git push --force` 仍屬受保護動作，必須先確認。

## 驗證標準

代理人至少應執行與變更相稱的驗證：

| 變更類型         | 最低驗證                                     |
| ---------------- | -------------------------------------------- |
| 文件／設定       | 格式檢查與相關設定驗證                       |
| 一般程式碼       | lint、型別檢查、受影響的單元測試             |
| 功能或使用者流程 | 前述檢查，加上建置與相關 E2E／煙霧測試       |
| 安全、部署或架構 | 前述檢查，加上設定／風險檢閱與明確的驗證缺口 |

若完整測試不可執行，必須先跑次佳驗證、說明原因與剩餘風險；不得把「未測試」說成完成。

## 回報格式

只在以下時機給簡短進度：開始、重大發現、受保護動作需要確認、或完成。完成回報必須包含：

- 已完成內容與主要檔案。
- 實際執行的驗證及結果。
- 已知風險或未驗證項目；若沒有，明確表示沒有已知遺留問題。

## 結束條件

只有同時滿足以下條件才結束：

1. 使用者要求的範圍已完成。
2. 對應驗證已通過，或驗證缺口已明確揭露。
3. 沒有可自行處理的失敗、待辦或回歸問題。
4. 沒有未獲確認的受保護動作。
