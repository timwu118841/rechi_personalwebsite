## Context

目前儲存庫只有 OpenSpec 設定，沒有既有應用程式可遷移。目標是建立單一作者、繁體中文、以法律經驗文章為核心的公開網站。作者不會操作 Git、Markdown 或部署指令，因此必須有可登入的視覺化 CMS 後台。使用者曾遇到 Payload CMS 回應慢與維運成本高的問題，因此公開流量不可依賴每次請求查詢 CMS 或資料庫。主要利害關係人是作者、搜尋引擎、一般讀者與站台維運者。

設計需同時兼顧：接近 Medium 的專注閱讀體驗、可持續的文章發布流程、搜尋引擎可見性、流量量測、Core Web Vitals，以及法律內容的免責與隱私風險。

## Goals / Non-Goals

**Goals:**

- 以靜態 HTML 為主要交付形式，公開閱讀不需要 CMS 或資料庫在線上回應。
- 提供受邀使用者可直接在瀏覽器登入、編輯、上傳媒體、預覽與發布的管理後台，不要求操作 Git。
- 建立型別安全、可驗證、具版本紀錄且能日後替換來源的文章模型與發布流程。
- 提供簡潔、現代、行動裝置優先且無障礙的閱讀與探索介面。
- 在初次上線就具備完整技術 SEO、RSS、站內搜尋與低負擔流量分析。
- 以自動化檢查守住效能、可索引性、內容 schema 與無障礙門檻。

**Non-Goals:**

- MVP 不提供公開註冊、讀者發文、複雜角色權限、訂閱付費牆、留言、會員、即時協作或個人化推薦。
- MVP 不導入 Payload CMS，也不部署長駐的 CMS／資料庫服務。
- 不複製 Medium 的品牌、視覺資產或完整社群功能；只採用其內容優先的資訊層級。
- 不把流量分析當成法律個案資料庫，也不蒐集文章讀者的敏感個資。

## Decisions

### 1. 使用 Astro 靜態公開頁與隔離的動態管理路由

以 Astro 建立網站，所有可索引頁面在建置階段產生 HTML；只有導覽互動、搜尋等必要功能才使用客戶端 JavaScript island。Keystatic 的 `/keystatic` 管理介面與驗證 API 使用 Vercel adapter 提供 on-demand server route，公開首頁、列表與文章頁仍是可由 CDN 快取的靜態檔案，不在公開請求路徑執行 CMS 查詢。

替代方案：Next.js 可以完成相同功能，但對這個單一作者內容站，預設的 React 與伺服器能力增加了不必要的執行面；純手刻靜態產生器則會自行承擔路由、內容 schema、圖片與生態整合成本。

### 2. 使用 Keystatic Cloud 做管理後台，內容仍版本化於 Git

Keystatic 提供 `/keystatic` 管理後台、登入、表單與所見即所得長文編輯器。採 Keystatic Cloud storage，讓受邀作者不需要 GitHub 帳號或 Git 操作；CMS 在背景將結構化資料與 Markdoc／Markdown 內容同步至 GitHub 儲存庫，保留版本歷史。MVP 限受邀管理者使用，不提供公開註冊或一般讀者發文。

必要欄位包含 `title`、`description`、`publishedAt`、`status`、`category`、`tags` 與正文；選填欄位包含 `updatedAt`、`cover`、`coverAlt`、`seoTitle`、`seoDescription`、`canonicalUrl`。slug 必須唯一且發布後維持穩定。管理介面提供圖片上傳、替代文字、草稿、格式化預覽、發布與下架操作；作者按下儲存或發布時，由 CMS 自動完成內容持久化與版本提交。

文章的內容類型與分類都不是寫死的選項，而是可由後台管理的 collection。內容類型用來區分「法律文章」「生活隨筆」「讀書筆記」等編輯用途；分類則用來組織「契約」「工作方法」等主題。作者可在管理介面新增或改名內容類型，並可新增、改名、撰寫分類說明、調整排序或暫停顯示分類，再由文章欄位選取兩者。

Keystatic 的 collection schema 與側欄模組不能由內容資料任意建立，因此後台固定提供「文章內容」「內容類型」「文章分類」三個模組，並將編輯與設定分組。日常擴充內容時只需新增內容類型或分類，不必修改 Keystatic schema 或前端程式碼；真正的欄位結構仍由程式碼定義，避免讓內容操作破壞資料契約。

GitHub 內容變更觸發 CI。建置時只輸出狀態為 published 且未來日期條件已滿足的文章，依發布日期排序，並產生分類、標籤、RSS、搜尋索引與 sitemap。內容存取集中在 repository/service 邊界，公開 UI 不直接耦合 Keystatic API 或檔案格式。

替代方案：Payload、Directus 等資料庫型 CMS 具有更完整的角色與內容關聯，但增加資料庫、API 與維運面；公開頁若誤用動態查詢也容易重現效能問題。純檔案式方案效能最佳，但不符合作者需要管理後台的需求。Keystatic 在這兩者間提供真正的 CMS 體驗，又能讓公開站從建置產物讀取；CMS 故障不影響既有文章的讀者。

### 3. 採內容優先的設計系統

建立小型 design token 集合，使用中性色、單一品牌強調色、清楚的字級階層、足夠留白與約 65–75 個字元的正文欄寬。頁面包含首頁精選、最新文章、文章列表、分類／標籤、文章頁、關於與搜尋。文章頁顯示閱讀時間、發布／更新日期、分類、標籤、分享連結、作者資訊與法律免責聲明。

字型以系統字型或自託管且子集化的字型為主，避免阻塞式第三方字型。互動元件支援鍵盤、可見焦點、降低動態效果偏好與足夠色彩對比。

### 4. 站內搜尋使用建置後的靜態索引

以 Pagefind 類型的建置後索引方案掃描輸出的 HTML，搜尋功能在瀏覽器載入小型分段索引，不建立搜尋伺服器。索引限於已發布文章，搜尋失敗時仍可透過分類、標籤與文章列表探索內容。

替代方案：Algolia 等託管搜尋對初期文章量過度複雜；客戶端一次載入所有文章全文會隨內容量線性增加下載成本。

### 5. SEO 由共用頁面契約集中產生

每個可索引路由透過共用 SEO 元件輸出唯一 title、description、canonical、Open Graph 與社群圖片資料。文章頁輸出 `BlogPosting`／`Article` JSON-LD，列表與導覽輸出適當的網站與麵包屑結構化資料。建置產生 `sitemap.xml`、`robots.txt` 與 RSS，草稿、預覽與錯誤頁不可進入 sitemap 且必須 noindex。

網址以穩定、可讀的 slug 為主；分類與標籤名稱另有顯示值與 URL-safe key。正式環境的 site URL 為必填設定，避免 canonical 使用錯誤網域。

### 6. 分析功能以可關閉的供應商介面整合

建立單一 analytics adapter；未設定 production site ID 時不載入任何追蹤程式。首選不使用 cookie、低 script 重量且支援自訂事件的隱私友善方案。若最終選用需要 cookie 或跨站識別的供應商，必須先取得同意才啟用。

MVP 量測頁面瀏覽、文章閱讀進度、外部連結點擊與站內搜尋，不傳送搜尋原文、文章內文、姓名、電子郵件、IP 或其他可識別法律個案的資料。另支援 Google Search Console 驗證與 sitemap 提交，但不把站長工具驗證碼寫死於元件。

### 7. 效能與品質採明確預算

公開內容頁初始客戶端 JavaScript gzip 預算為 75 KB；非互動文章正文不 hydration。圖片在建置時產生適當尺寸與現代格式，固定寬高以避免版面位移，首屏主圖不延遲載入，其餘圖片 lazy-load。正式站以 Lighthouse CI 與內容檢查驗證 Performance、Accessibility、Best Practices 至少 95，SEO 100；真實使用者目標為 LCP ≤ 2.5 秒、INP ≤ 200 毫秒、CLS ≤ 0.1（第 75 百分位）。

分析與搜尋程式不得阻塞主要內容呈現；效能檢查在合併與發布前執行。

### 8. 法律內容的安全界線

全站與文章頁必須清楚表達內容是一般資訊與個人經驗分享，不構成針對個案的法律意見或律師關係。文章發布前的檢查清單要求移除不必要的姓名、聯絡方式、案號與其他可識別個資；需要引用時由作者確認合法性與匿名化程度。

## Risks / Trade-offs

- [Keystatic Cloud、GitHub 或管理函式暫時故障時作者無法編輯] → 公開站仍由前一次靜態產物提供服務；內容保留在 Git，可於服務恢復後繼續編輯，並保有匯出／更換 CMS 的路徑。
- [靜態建置讓內容更新需要重新部署] → 以 CI 在內容合併後自動建置與部署；單一作者與初期文章量下，建置成本可控。
- [第三方分析可能影響隱私與效能] → 預設可停用、延後載入、限制事件欄位，並以效能預算阻止回歸。
- [法律文章可能包含敏感或可識別資訊] → 加入發布檢查清單、全站免責聲明與最小資料原則；內容法律責任仍由作者審核。
- [Lighthouse 實驗室分數不等同真實使用者體驗] → 同時收集匿名化 Web Vitals／平台數據，以上線後的第 75 百分位趨勢校正。
- [SEO 成效需要時間且不保證排名] → 確保技術可索引性、內容品質、穩定網址與站長工具監控，不承諾特定關鍵字名次。

## Migration Plan

1. 建立 Astro 專案、Keystatic schema、本機管理後台、範例文章與公開路由，確認管理端儲存可產生有效內容檔。
2. 連接 Keystatic Cloud／GitHub、限制受邀使用者登入，並確認發布動作會觸發建置與 preview。
3. 完成設計系統、文章閱讀、探索、搜尋、SEO、RSS 與分析 adapter。
4. 在 CI 加入型別、內容 schema、管理權限、單元／整合、無障礙、連結、SEO 與 Lighthouse 檢查。
5. 設定正式網域、CMS secrets、分析 site ID、站長工具驗證與 sitemap，部署到可同時提供靜態 CDN 與受保護管理函式的平台。
6. 先以 noindex 的 preview 環境驗收管理與發布流程，再開放正式站索引；若上線失敗，回滾到前一個靜態部署版本。

## Open Questions

- 正式網域、站名、作者顯示名稱、品牌色與作者照片尚待提供；實作先使用集中設定與可替換預設值。
- 分析供應商尚未指定；實作不得讓供應商選擇耦合核心頁面。部署平台採 Vercel，但公開內容仍維持預先產生的靜態輸出。
- MVP 預設最多三位受邀後台使用者；若需要更多作者、審稿角色或分級權限，需另行擴充 CMS 權限模型。
